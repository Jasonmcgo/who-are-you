# CC-138.1 — Block binary contamination fingerprints from the cross-signal lift

> Point-release follow-up to CC-138, exactly like CC-134.1 followed CC-134.
> Surgical: one constant + a regression assertion. No new behavior beyond the gate.

## Context

CC-141 made the CC-097B cross-signal lift refuse to raise confidence to "high"
when the typing carries a contamination flag. The gate is an allowlist of
blocking reasons in `lib/identityEngine.ts` (`attachCrossSignalDriverInference`,
~L2553):

```ts
const blockingReasons: ReadonlyArray<string> = [
  "ns-valence",
  "judging-cooccurrence",
  "thin-floor",
];
```

CC-138 introduced two **new** low-confidence reasons for binary sessions:

- `binary-attitude-violation` — a same-attitude pick on either axis (e.g. Ni+Si
  or Ti+Fi). This is the single strongest "this typing is contaminated" signal
  the binary format produces.
- `binary-dominance-ambiguous` — the dominance ordering pick is missing or
  doesn't match either parent attitude pick.

These are **not** in `blockingReasons`. So today the cross-signal lift can still
fire on dominant agreement and harden a contaminated binary typing to "high" —
the exact failure mode CC-141 was built to close, re-opened for binary sessions.
A contamination fingerprint should be the *least* liftable state, not a liftable one.

## Task (in `lib/identityEngine.ts`)

Add both new reasons to the `blockingReasons` allowlist:

```ts
const blockingReasons: ReadonlyArray<string> = [
  "ns-valence",
  "judging-cooccurrence",
  "thin-floor",
  "binary-attitude-violation",   // CC-138.1 — binary contamination fingerprint
  "binary-dominance-ambiguous",  // CC-138.1 — binary dominance unresolved
];
```

That is the entire change to engine behavior. Do **not** alter the lift's score/gap
floors, the §D clarifier eligibility (CC-138 already wired the binary reasons
there), the binary resolver, or anything else. The clarifier still fires on these
reasons (parallel `confidenceLowReasons` channel) — this CC only stops the lift
from *overriding* them.

## Regression assertion

Add an assertion to the existing CC-141 lift audit (or the
`functionVoiceBinary` audit, whichever already constructs a binary session with a
violation) proving: a binary session that (a) has `binary-attitude-violation` in
`confidenceLowReasons` AND (b) whose cross-signal inferred driver agrees with the
dominant AND (c) clears the score/gap floors — **stays "low"** (lift blocked).
Before this CC that case would lift to "high". Show the before/after.
