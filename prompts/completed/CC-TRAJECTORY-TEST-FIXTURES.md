# CC-TRAJECTORY-TEST-FIXTURES — Test Fixtures for Trajectory Model Refinement

**Origin:** `docs/canon/trajectory-model-refinement.md` §19. Per Jason + Clarence canon (2026-05-10), the three-phase trajectory model refinement requires test fixtures that exercise each new derivation cleanly. This CC adds those fixtures so Phase 1, Phase 2, and Phase 3 audits have stable reference cases.

**Method discipline:** Pure data work. No engine changes, no prose changes, no LLM prompt changes. Adds fixture JSON files only. Validation: each new fixture builds cleanly through the existing engine (no schema breakage), and the existing cohort audits continue to pass.

**Scope frame:** ~1-2 hours executor time. CODEX-scale (data-only) but with editorial judgment in fixture design — each fixture must hit specific signal combinations to exercise the canon's test cases.

**Cost surface:** Zero. No LLM calls. No cohort cache regeneration required (new fixtures don't yet have cached prose; that will be generated by future CCs after Phase 1/2/3 land).

**Parallel-eligible:** This CC touches `tests/fixtures/` only. Zero file conflicts with CC-3CS-STRENGTH-MIX-AXIS-ALIGNMENT (Phase 1) or any other engine-track CC. Can fire simultaneously with Phase 1.

## Embedded context (CC executor environments don't see Cowork memory)

### Canon reference

Full canon at `docs/canon/trajectory-model-refinement.md`. §19 specifies five required test fixtures:

1. **High Goal / High Cost Builder** — validates CostStrength couples to Goal score under Builder shape
2. **High Soul / High Coverage Caregiver** — validates CoverageStrength couples to Soul score under Caregiver shape
3. **High Compliance / Low Grip Wise-Risk Steward** — validates ComplianceStrength produces high Aim, classification Open-Handed Aim (Phase 3)
4. **High Stakes / Low DefensiveGrip Responsible Leader** — validates StakesLoad ≠ DefensiveGrip; Grip does not spike just because stakes are high
5. **Jason fixture revalidation** — the existing fixture; CC-3CS Phase 1 audit asserts CostStrength ≥ 80 against this fixture

### Why fixtures are needed BEFORE engine work lands

Two reasons:

1. **Audit assertions reference fixture paths.** The CC-3CS-STRENGTH-MIX-AXIS-ALIGNMENT audit's substrate-alignment assertion (`abs(CostStrength - goalScore) <= 5` for every fixture) needs fixtures that exercise high-Cost, high-Coverage, and high-Compliance shapes. Adding fixtures parallel-with-Phase-1 means Phase 1's audit can reference them at landing time.

2. **Each phase's validation depends on having representative shapes.** Phase 2's Aim formula validation needs a high-Compliance-low-Grip fixture to assert Open-Handed Aim. Phase 3's label refinement needs a 50°-band fixture (Goal-led Presence) AND a 32°-band fixture (Productive but under-integrated). Without fixtures, audits become hand-waving.

### Fixture design constraints

Each fixture should:

- Be schema-compatible with existing fixture format (same JSON shape as `ocean/24-si-precedent-keeper.json` and similar)
- Hit the target signal combination cleanly (high Goal means high values on Goal-driver signals; high Coverage means high values on Soul-driver signals)
- Be plausibly real — represent a recognizable shape, not a synthetic edge case (unless explicitly an edge-case test)
- Carry an age demographic so age-band classification fires (per CC-AGE-CALIBRATION)
- Not duplicate an existing fixture's shape — each new fixture should test something the cohort doesn't already cover
