## Context ‚Äî two defects

**Defect 1 ‚Äî both typing formats are presented.** CC-138 added the six Q-TB-*
binaries *after* Q-T8 but left Q-T1‚ÄìQ-T8 (the 4-way "Four Voices" rank cards) in
the presented question sequence. `app/assessment/page.tsx` renders
`questions[current]` straight from `data/questions.ts` in array order, so a NEW
user now answers BOTH the legacy 8 ranking cards AND the 6 binaries. The legacy
ranking answers are then ignored anyway (`hasBinarySignals` ‚Üí binary resolver
wins), so the Q-T cards are redundant, confusing, and lengthen the test. The
intended end-state: new users see ONLY the binary 2-box format for typing.

**Defect 2 ‚Äî Back is broken across auto-advance questions.** Three effects in
`app/assessment/page.tsx` auto-advance FORWARD on entry, regardless of nav
direction:
- conditional `freeform`/`forced` with an answered `render_if_skipped` target (~L283),
- `ranking_derived` cascade-skip when `derivedItems === null` (~L307),
- `multiselect_derived` cascade-skip when items are null (~L330).

The Back button does `setCurrent(current - 1)` (~L906) with no backward-skip. So
pressing Back onto any auto-skip question fires its forward auto-advance and
bounces the user right back where they were ‚Äî Back appears dead. (Pre-existing,
but CC-138's extra questions made it easy to hit.)

## Tasks

### Part A ‚Äî Retire Q-T1‚ÄìQ-T8 from the presented sequence (keep them resolvable)

The hard constraint: legacy saved sessions and the cohort fixtures store Q-T
ranking answers and MUST still resolve via the existing legacy path
(`aggregateLensStack`'s pre-CC-138 branch). So the Q-T question DEFINITIONS and
their item‚Üísignal maps must remain available to the resolver/parsers ‚Äî only their
*presentation* in the live flow is removed.

Pick and document a clean mechanism, e.g.:
- Add `legacy: true` to the Q-T1‚ÄìQ-T8 objects (and any Q-T-only derived children
  the binaries replace) in `data/questions.ts`; export the full set as
  `allQuestions` and the presented set as `questions = allQuestions.filter(q => !q.legacy)`.
- Any lookup that must see legacy defs (signal/item maps for parsing saved Q-T
  answers) reads `allQuestions`; the assessment flow and anything that should
  reflect the live sequence reads `questions`.

Audit every `questions.find(...)` / `questions.findIndex(...)` / `questions.length`
use (in `app/assessment/page.tsx` and elsewhere) and route each to the correct
list: presented-flow logic ‚Üí `questions`; saved-answer parsing/resolution ‚Üí
`allQuestions`. Do not break `Q_I1_INDEX` or the second-pass boundary.

Check for orphaned derived children: if any `ranking_derived` /
`multiselect_derived` question's parents are among the retired Q-T set, retire the
child too (it's superseded by Q-TB-PERC-ORDER / Q-TB-JUDG-ORDER). Do NOT retire
the Q-TB derived orderings.

### Part B ‚Äî Direction-aware Back (skip backward over auto-advance questions)

Make Back move to the nearest previous *presentable* question instead of bouncing
forward. Recommended approach (robust to consecutive auto-skip questions): a
`navDirectionRef` (`"forward" | "back"`), set by `onContinue`/`handleSkip`
(forward) and the Back handler (back). The three auto-advance effects, when
`navDirectionRef.current === "back"`, should RETREAT (move to the previous index)
instead of calling `advanceFromIndex` ‚Äî i.e. add a `retreatFromIndex(idx)` mirror
of `advanceFromIndex` that decrements and skips backward, stopping cleanly at
index 0 (if index 0 is itself auto-skip, leave `onBack` undefined / Back disabled).

Equivalent acceptable alternative: have `onBack` itself compute the previous
presentable index (walking backward, applying the same skip predicates the three
effects use) and `setCurrent` to it directly, leaving the forward effects
untouched. Either is fine; pick one, keep the skip predicate single-sourced so it
can't drift from the effects' conditions.

Whichever you choose: Back must traverse the WHOLE flow correctly ‚Äî including
backing across the Q-TB binaries, the dominance orderings, derived cards, and the
pre-Keystone second-pass boundary ‚Äî without ever bouncing forward.
